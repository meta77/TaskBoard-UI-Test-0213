# 実装計画 (Implementation Plan)

## 目的
ドラッグ＆ドロップ（DnD）操作において、以下の2つのエッジケース（境界条件）に対してアプリがクラッシュしたりデータが破壊されたりしないよう、統合テスト（`app.test.js`）を追加します。テストが失敗する場合はロジックを修正して対応します。

1. **何も選ばずにドロップ（`onDragStart` が呼ばれていない状態での `onDrop`）**
2. **自分自身の上にドロップ（元の位置と完全に同じ場所にドロップ）**

## 変更内容

### 1. `app.test.js` (統合テストの追加)
新しい2つのテストケースを追加します。

- **[ADD] テストケース 1: 「何も選ばずにドロップした場合」**
  - アクション: `app.onDrop` を直接呼び出します（`draggedTaskId` をセットせずに）。
  - アサーション（期待値）: `app.tasks.value` の内容や順序が一切変更されていないこと。エラーが発生しないこと。

- **[ADD] テストケース 2: 「元の位置と同じ場所（自分自身の上）にドロップした場合」**
  - アクション: `onDragStart` にあるタスクID（例：`SYS-101`）を渡し、その後同じタスクIDを `targetTaskId` として `onDrop` に渡します。
  - アサーション（期待値）: タスクの順序やステータスが元の状態から変更されていないこと。

### 2. `logic.test.js` (純粋関数のテスト追加)
ビジネスロジックである `moveTaskInArray` 関数自体が、「自分自身を指定されたターゲット」として受け取った時の振る舞いを保証するテストを追加します。

- **[ADD] テストケース: 「移動対象とターゲットが同じIDの場合、元の配列を変更せずに返すこと」**

### 3. `logic.js` (ロジックの修正)
上記のテスト2（自分自身へのドロップ）を受けた際、現在地の `moveTaskInArray` は一度タスクを配列から削除し、その後で挿入先を探そうとします。自分が見つからないため、意図せず配列の末尾にタスクが移動してしまうバグを含んでいます。これを修正します。

- **[MODIFY] `moveTaskInArray`**
  - 関数の先頭に早期リターンのガード句を追加します。
  - `if (taskId === targetTaskId) return tasks;` を追加し、無駄な処理や配列破壊を防ぎます。

## 検証・テスト計画 (Verification Plan)
1. 上記のテストを追加した直後に `npm test` を実行し、既存のテストに加え新設した「ドラッグ＆ドロップのエッジケーステスト」がPassすることを確認します。
2. これにより、未選択時のドロップ例外や、自分自身へドロップした時の意図せぬ移動バグが完全に防がれたことを継続的に保証します。
